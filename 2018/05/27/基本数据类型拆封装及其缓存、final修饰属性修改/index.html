基本数据类型拆封装及其缓存、final修饰属性修改

学习背景

public static void main(String[] args) {
    Integer a = 1, b = 2;
    System.out.println("before : a=" + a + ", b=" + b); //before : a=1, b=2
    swap(a, b);
    System.out.println("after  : a=" + a + ", b=" + b); //after  : a=2, b=1
}

public static void swap(Integer i1, Integer i2) {
    try {
        Field field = Integer.class.getDeclaredField("value");
        field.setAccessible(true);
        Integer tmp = new Integer(i1);
        field.set(i1, i2.intValue());  
        //field.set(i2, tmp);          //-------------------------1
        field.set(i2, tmp.intValue()); //-------------------------2
    }catch (NoSuchFieldException e) {
        e.getStackTrace();
    }catch (IllegalArgumentException e) {
        e.getStackTrace();
    }catch (IllegalAccessException e) {
        e.getStackTrace();
    }

上面的就是一个刚学java的人问我的一个问题，main方法中是问题给出的，需要自己写一个swap方法来交换a与b的值，然后下面的swap是他写的方法，结果他发现使用上文中1的方式可以正确交换两个值，而用2的方式输出的a以及b都为2，向我求问这里的原因。很丢人的事，我花了好长时间才给他解答，原因就是其实我也一下子没理解为什么会这样。

题目分析

首先拿到这个问题，我先想了下这个题目考的是什么，首先这里用于交换用的是包装类，有关于引用传递的。然后查看Integer中存放int值的value字段的限定符是用final修饰的，所以这边也只能用反射去处理。其实我最开始想到的就是这两点，而且看了他的解答之后我也只想到了这两点，其实还想到了拆封装，但我并没有看到在方法中有拆封装的地方，其实这也就是上面所述问题的根本原因。

问题解析

乍一看1与2都没什么毛病，1传的是Integer包装数据，2传的是int基础类型，结果两者产生了差别，其实这两者平时都是一样的，但是Field的set方法的定义是public void set(Object obj, Object value),第二个参数也是个Object类型，而2中传的是个int基础类型，这边就会多一个步骤，基础类型的封装。要知道java中的8个基础类型都有各自的包装类，当需要转换为对象时，就会产生一个封装的步骤。比如long就会转换为Long,int就会转换未Integer等等。很多人可能认为，这里的封装就是使用new Integer(int)的方式进行的，如果真是这样，交换得到的值应该就是正确的。问题的关键就在这里，这里的封装并不是使用构建方法封装，具体的方式可以使用javap -v ***.class的方式反编译class，得到class的字节码内容。

上述就是swap方法的字节码内容，而这边需要重点关注的就是序号为8的那一行，在注释中可以看到，这里调用了Integer.valueOf方法，原来代码中并没有使用此方法，可想而知，其实封装调用的就是Integer.valueOf方法，那么再来看下这个封装方法里的内容

public static Integer valueOf(int i) {
    if (i >= IntegerCache.low && i <= IntegerCache.high)
        return IntegerCache.cache[i + (-IntegerCache.low)];
    return new Integer(i);
}

在调用构造方法生成Integer之前还有个判断，判断是在-127~128之内，如果是的话，是从缓存中获取然后返回。Integer自己内部预先构造生成了-127~128之间的Integer对象存在缓存中，其实也就是说，如果由反射生成的Integer且数量值n在-127~128之内，则返回的对象就从这个cache数组中取第127+n个对象。由此内容就可以解决本次的这个问题了。

  1. 在main方法中首先通过包装生成了1和2的包装对象，返回了缓存数组中第128和第129个对象
  2. field.set(i1, i2.intValue());行中将第128个对象的值修改为了2，此时缓存数组中第128以及第129个对象的值均为2
  3. 如果使用field.set(i2, tmp);,此时传递的是Integer对象，且符合public void set(Object obj, Object value)方法定义，不需要封装，只是在反射的内存会有个拆装取出值1赋予给对象b，此时得到的结果正确
  4. 如果使用field.set(i2, tmp.intValue())，此时传递是基础类型int，不符合public void set(Object obj, Object value)方法定义，需要封装，调用Integer.valueOf方法，取出原来应该值为1位置上的缓存对象，也就是第128个对象，但此时该对象的值因为步骤2的关系，值已经变成了2，所以这一步最后做的是将2赋值给对象a，因此，最后得到的结果是两个值均为2

也就是说，其实如果a跟b其中一个的值在-127~128之外就不会有这样的问题，或者a跟b是使用构造方法生成的，而不是通过封装生成的，也不会有这样的问题。

####问题反思
其实这道题的内容在正常的开发中是不可能出现的，如果真的出现了那只能说明开发人员在瞎写。首先对于属性为final声明的变量，理应就是在初始化之后不可更改的，虽然java提供了反射的方式可以越过这些校验来修改，但正常情况下应该遵守代码规范。其次就算真的修改了，会对应用后面的内容产生不可预知的问题。比如我在之后的代码中声明一个Integer c = 1;此时如果使用这个对象c的话，它真正代表的值却是2，这也是由于拆装以及包装引起的，这已经违背开发人员的意愿了，并且这个问题将在之后的应用中一直存在，这个问题是非常恐怖的。但是对于Integer以及Long的缓存这个问题，日常开发中时常会碰到Long == Long的情形，这个情形一般是一些初学者经常会犯且很难察觉的问题。


撇开存在性，这又是一个非常好的锻炼个人基础知识的问题，此问题涉及了基本类型的拆封装，Integer缓存（Long也有），反射，final属性修改，值传递和引用传递等知识点，如果作为面试官，这也不失为一道好题。



ps.对应final属性有时其实无法修改，那就是在声明时就初始化的内容基本类型跟String,比如final int a = 1;，即使使用反射修改了，使用get方法获取发现还是原来的值，其中的奥妙后面的文章再说，留给各位思考。
